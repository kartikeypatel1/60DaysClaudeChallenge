MODULE 01Foundational

What Is a Tree?

Start here with the vocabulary and shapes every later module depends on.

1.1 The core idea

A tree is a hierarchical structure made of nodes connected by edges, with no cycles. One node is the root (no parent); every other node has exactly one parent and zero or more children. A node with no children is a leaf.

Analogy
Think of a company org chart. The CEO is the root — nobody reports to them. Each manager has direct reports (children), and individual contributors with no direct reports are leaves. You can trace a line of command from any employee straight back to the CEO — that's exactly what "path to root" means in a tree.
Root → internal nodes → leaves. Height = longest path from root to a leaf (here, 2).

1.2 Terminology cheat sheet

TermMeaning
RootThe single top node; has no parent
Parent / ChildDirect connection one level apart
LeafA node with zero children
Depth of a nodeNumber of edges from root to that node
Height of a treeNumber of edges on the longest root-to-leaf path
SubtreeAny node plus all its descendants, treated as its own tree

1.3 Binary trees and their shapes

A binary tree restricts every node to at most two children, conventionally called left and right. Shape matters a lot for performance:

Full

Every node has 0 or 2 children — never exactly 1.

Complete

All levels filled except possibly the last, which fills left-to-right. This is what powers array-based heaps.

Perfect

All internal nodes have 2 children AND all leaves sit at the same depth.

Balanced

For every node, its left and right subtree heights differ by at most 1. Guarantees O(log n) operations.

1.4 How trees live in memory

Two common representations:

// Linked representation (most common for BSTs)
class Node {
  constructor(value) {
    this.value = value;
    this.left = null;
    this.right = null;
  }
}

// Array representation (common for complete binary trees / heaps)
// For index i: left child = 2i+1, right child = 2i+2, parent = floor((i-1)/2)
const tree = [8, 3, 12, 1, 6, null, 15];
Common Misconception
"A tree with N nodes always has height log₂(N)." False — that's only true if the tree is balanced. A tree where every node has exactly one child (a "chain") has height N−1, no better than a linked list.

Key Takeaways

  • Trees encode hierarchy: one root, no cycles, each node has one parent.
  • Height and depth are edge-counts, not node-counts — an easy off-by-one trap.
  • Tree shape (full / complete / perfect / balanced) determines what guarantees you get.
  • Shape is not automatic — a naive BST can degrade into a linked list.
Quick Exercise

A binary tree has 7 nodes and every internal node has exactly 2 children, with all leaves at the same depth. What is its height?

MODULE 02Core Mechanics

Binary Search Trees

Add one rule to a binary tree and it becomes a search structure: for every node, everything in its left subtree is smaller, everything in its right subtree is larger.

Analogy
It's the "guess the number, higher or lower" game turned into a data structure. Every comparison eliminates one whole half of the remaining candidates — exactly like binary search on a sorted array, except the tree doesn't need contiguous memory to do it.

2.1 Search

function search(node, target) {
  if (node === null) return null;           // not found
  if (target === node.value) return node;   // found it
  if (target < node.value) return search(node.left, target);
  return search(node.right, target);
}

Each comparison discards an entire subtree. In a balanced tree of N nodes, that's O(log n) comparisons to find anything.

2.2 Insertion

function insert(node, value) {
  if (node === null) return new Node(value);
  if (value < node.value) node.left = insert(node.left, value);
  else if (value > node.value) node.right = insert(node.right, value);
  return node; // duplicates ignored here
}

2.3 Deletion — the three cases

Deletion is the operation that trips people up because there are three distinct shapes to handle:

CaseNode has...Fix
1No children (leaf)Simply remove it
2One childReplace the node with its one child
3Two childrenReplace its value with its inorder successor (smallest value in right subtree), then delete that successor
function deleteNode(node, value) {
  if (node === null) return null;
  if (value < node.value) node.left = deleteNode(node.left, value);
  else if (value > node.value) node.right = deleteNode(node.right, value);
  else {
    if (node.left === null) return node.right;   // case 1 & 2
    if (node.right === null) return node.left;    // case 1 & 2
    let succ = node.right;                          // case 3
    while (succ.left) succ = succ.left;
    node.value = succ.value;
    node.right = deleteNode(node.right, succ.value);
  }
  return node;
}

2.4 Traversals

Four ways to visit every node — each gives a different view of the same tree:

Inorder (L, Root, R)

Visits nodes in sorted order for a BST. The single most important traversal to know.

Preorder (Root, L, R)

Visits the root first — useful for copying/serializing a tree's structure.

Postorder (L, R, Root)

Visits children before the parent — useful for safely deleting a tree bottom-up.

Level-order (BFS)

Visits level by level using a queue, not recursion — shows the tree's actual shape.

Inorder: 2, 3, 4, 5, 8 (sorted!) · Preorder: 5, 3, 2, 4, 8 · Postorder: 2, 4, 3, 8, 5 · Level-order: 5, 3, 8, 2, 4

2.5 Complexity at a glance

OperationBalanced BSTWorst case (chain)
SearchO(log n)O(n)
InsertO(log n)O(n)
DeleteO(log n)O(n)
Traversal (all nodes)O(n)O(n)
Common Misconception
"Deleting a node with two children just removes it and reconnects the children directly." No — you must promote a successor (or predecessor) to preserve the BST property. Reconnecting children directly usually breaks ordering.

Key Takeaways

  • The BST invariant — left < node < right — must hold at every node, not just the root.
  • Inorder traversal of a BST always yields sorted output.
  • Deletion has exactly three cases; the two-children case is the one to memorize.
  • All the nice complexity numbers assume balance — nothing enforces that automatically yet (Module 3 fixes this).

MODULE 03Practical Application

Balance: Why Your BST Can Betray You

An unbalanced BST is a trap that only springs at scale — it works fine in every small test case, then quietly becomes O(n) in production.

3.1 The degenerate case

Insert 1, 2, 3, 4, 5 into an empty BST in that order. Every new value is larger than everything before it, so it always goes right. You get a right-leaning chain — structurally a linked list wearing a tree's clothing.

Sorted input → a chain. Height = 4 for 5 nodes, instead of ~2. Search degrades to O(n).
Analogy
Balance is like keeping a bookshelf sorted vs. keeping it sorted and evenly spread across shelves. A sorted single shelf still lets you scan-search in order, but a well-spread multi-shelf layout lets you jump straight to the right area — that "jump straight there" behavior is what balance protects.

3.2 How self-balancing trees fix this

Self-balancing trees (AVL trees, Red-Black trees) perform extra bookkeeping on every insert/delete to guarantee height stays O(log n).

AVL Trees

Track a balance factor (height of left − height of right) per node, must stay in {−1, 0, 1}. Violations trigger rotations — single or double — that restructure locally in O(1) per fix.

Red-Black Trees

Color nodes red/black under strict rules (no two reds in a row, equal black-height on every path). Looser balance than AVL, but cheaper rebalancing — this is what most language standard libraries (e.g. C++ map, Java TreeMap) actually use.

3.3 Rotation, conceptually

A right rotation takes a left-heavy imbalance and pivots the left child up to become the new local root, pushing the old root down to be its right child. It's a constant-time pointer rearrangement that fixes height without touching the BST ordering property.

// Right rotation around node y (y.left = x)
function rotateRight(y) {
  const x = y.left;
  y.left = x.right;
  x.right = y;
  return x; // x is the new subtree root
}

3.4 Real-world uses

  • Database indexes — though most use B-trees (a wide, shallow generalization) rather than plain BSTs, for disk-page efficiency.
  • Language runtime maps/sets — Red-Black trees back C++'s std::map and Java's TreeMap/TreeSet.
  • Filesystems — directory hierarchies are literal trees; some filesystems use balanced trees internally for metadata.
  • Autocomplete / range queries — inorder traversal + BST search makes "give me everything between X and Y" efficient.
Practical Exercise

Use the BST builder from Module 2's mindset (mentally, or scroll up to reuse the widget): if you insert 10, 20, 30, 40, 50 in that order, what will the resulting tree's height be, in edges?

Common Misconception
"Random insert order is dangerous too." Actually, random insertion order tends to produce a reasonably balanced tree on average — O(log n) expected height. It's sorted or near-sorted input that's the real danger, which real-world data (timestamps, IDs) often is.

Key Takeaways

  • A plain BST gives no shape guarantee — sorted input degenerates it into a linked list.
  • AVL trees enforce strict balance via rotations; Red-Black trees enforce looser balance more cheaply.
  • Rotations fix height in O(1) without breaking the BST ordering invariant.
  • This is exactly why production key-value structures use self-balancing trees, not plain BSTs.

MODULE 04Mastery

Interview-Grade BST Problems

Four classic problems that show up constantly — each one is a direct application of everything in Modules 1–3.

4.1 Validate a Binary Search Tree

A frequent bug: checking only node.left.value < node.value < node.right.value locally. That misses violations further down — a right-grandchild could still be smaller than a far-left ancestor. The fix is to carry a valid (min, max) range down the recursion.

function isValidBST(node, min = -Infinity, max = Infinity) {
  if (node === null) return true;
  if (node.value <= min || node.value >= max) return false;
  return isValidBST(node.left, min, node.value)
      && isValidBST(node.right, node.value, max);
}

4.2 Lowest Common Ancestor (LCA) in a BST

Because of the ordering property, LCA in a BST is simpler than in a general tree — no need to search both subtrees.

function lca(node, p, q) {
  if (p < node.value && q < node.value) return lca(node.left, p, q);
  if (p > node.value && q > node.value) return lca(node.right, p, q);
  return node; // p and q split here — this is the LCA
}
Analogy
It's like finding where two people's paths from the root first diverge. Walk down together while both targets are on the same side; the moment they'd go different directions, you're standing at their meeting point.

4.3 Convert a sorted array into a balanced BST

Always pick the middle element as the root — this guarantees perfect balance in one pass.

function sortedArrayToBST(arr, lo = 0, hi = arr.length - 1) {
  if (lo > hi) return null;
  const mid = Math.floor((lo + hi) / 2);
  const node = new Node(arr[mid]);
  node.left = sortedArrayToBST(arr, lo, mid - 1);
  node.right = sortedArrayToBST(arr, mid + 1, hi);
  return node;
}

4.4 Check if a tree is height-balanced

function checkBalanced(node) {
  function height(n) {
    if (n === null) return 0;
    const lh = height(n.left);
    if (lh === -1) return -1;
    const rh = height(n.right);
    if (rh === -1) return -1;
    if (Math.abs(lh - rh) > 1) return -1; // signal: unbalanced
    return Math.max(lh, rh) + 1;
  }
  return height(node) !== -1;
}

The trick: return -1 as a sentinel the moment imbalance is detected anywhere, so you don't waste time computing heights for a tree you already know fails.

Comparisons: BST vs. Alternatives

StructureSearchSorted iterationNotes
Balanced BSTO(log n)O(n), naturally in orderBest when you need both search and ordered traversal
Hash TableO(1) avgNot supported directlyFaster lookups, but no ordering, no range queries
Sorted ArrayO(log n)O(n), trivially in orderFast search, but O(n) insert/delete (shifting)
Linked ListO(n)O(n) only if pre-sortedFast insert at known position, poor search
Common Misconception
"If I just need fast lookups, a BST is a fine default." Usually a hash table wins for pure lookups. Reach for a BST specifically when you also need ordering, range queries, or "find nearest" style operations — that's its actual edge over a hash table.

Key Takeaways

  • Validating a BST requires a range check, not just a local parent-child comparison.
  • BST ordering makes LCA a simple directional walk, unlike general trees.
  • Building from a sorted array by picking the middle element guarantees balance for free.
  • Choose a BST over a hash table specifically when you need order or range queries, not just lookup speed.

Cheat Sheet

Complexities (balanced)

Search / Insert / Delete: O(log n)
Traversal: O(n)
Worst case (chain): O(n) for all

Traversal order

Inorder → sorted
Preorder → serialize/copy
Postorder → safe delete
Level-order → BFS by depth

Deletion cases

0 children → remove
1 child → splice child up
2 children → swap with inorder successor, then delete successor

Balance tools

AVL → strict, uses rotations
Red-Black → looser, cheaper rebalancing
Sorted array → mid element gives free balance

Summary Notes

  • A BST is a binary tree with one invariant: left < node < right, everywhere.
  • That invariant is what turns tree search into O(log n) — but only if the tree stays balanced.
  • Nothing about a plain BST enforces balance; sorted-ish input will degrade it toward O(n).
  • AVL and Red-Black trees exist purely to make that guarantee automatic via rotations.
  • Master four building blocks: search, insert, delete (3 cases), and the four traversals — everything else in this module is a composition of those.
