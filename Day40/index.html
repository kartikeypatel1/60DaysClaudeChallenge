System Prompt Design

GapFinder's "brain" is a single system prompt that forces every model turn to return strict JSON matching one fixed schema: score, feedback, model_answer, gap_tags, topic, next_question, interview_complete. This turns a naturally freeform conversational task into something the UI can render deterministically — no fragile text-parsing, no regex-scraping a chat transcript.

Scope is intentionally narrow: technical questions only, tailored to the resume and JD, no reused proprietary questions, no behavioral questions unless asked. The prompt encodes a scoring rubric (1–3 major gaps, 4–6 partial, 7–8 solid, 9–10 interview-ready) so scores stay consistent across a whole session instead of drifting.

Edge cases are handled explicitly in-prompt rather than in app code: empty/off-topic answers still get scored conservatively and the interview continues; a gibberish resume still produces a sensible fresher-level question instead of a refusal; meta-questions from the candidate are treated as their answer rather than breaking the JSON contract.

Conversation Architecture

The app is stateless between page loads and sends the full message history on every API call (resume + JD + every Q&A pair so far), since the Claude API itself holds no memory. Each assistant turn's raw JSON reply is stored back into the message array, so the model can see its own prior questions and avoid repeating topics.

Question count is controlled conversationally — the app tells the model "this is question N of TOTAL" each turn, and the model sets interview_complete: true once done, rather than the client trying to cut off the model mid-flow.

UI Decisions

  • Score circles are color-coded (red / yellow / green) for instant scanability without reading numbers.
  • The model answer is collapsed by default so candidates try to self-assess against their own answer first before peeking.
  • A running average-score readout in the progress bar gives a live sense of trajectory, not just a single end score.
  • Dark near-black theme with amber accents and monospace type keeps the "technical interview" feeling closer to a terminal than a form.

Ideas For Future Extensions

  • Tools: a code-execution sandbox tool so DSA answers can be run against test cases instead of just judged as text; a resume-parsing tool to accept PDF/DOCX uploads directly.
  • Memory: persist gap tags across sessions (with user consent) so a returning candidate gets questions targeted at their previously weak areas first.
  • Multi-step workflows: an adaptive difficulty engine that branches into a full system-design round after strong DSA performance; a "second pass" mode that re-asks only the questions scored below 6.
  • Voice mode: spoken answers with latency-aware transcription, to simulate real interview pressure more closely than typing.
